# SaaS Revenue Intelligence

An MCP server that estimates the Annual Recurring Revenue (ARR) of any SaaS company from its domain. Deployed on [Context Protocol](https://ctxprotocol.com) as a paid tool — you query it with a domain, it returns a calibrated ARR estimate with confidence score and signal breakdown.

**Mean error: 1.35x** across 60 public SaaS companies with known ARRs.

---

## How It Works

Five signals are fetched concurrently and fed into a Ridge regression model trained on S-1 and public ARR data:

| Signal | Source | What it measures |
|--------|--------|-----------------|
| Headcount | Crustdata → PDL → SerpAPI → LinkedIn | Revenue capacity (primary signal) |
| Hiring velocity | Greenhouse / Lever / Ashby / Workable | Growth rate |
| Web traffic | Tranco top-1M rank | Brand reach |
| Review momentum | G2 / Trustpilot | Customer acquisition |
| Pricing tier | Scraped pricing page | ACV estimate |

Results are cached in Redis for 24 hours. Cold fetches take 10–25s. Cached responses return in under 3s.

## Setup

**Requirements:** Python 3.10+, Redis (via Upstash)

```bash
git clone https://github.com/your-username/saas-revenue-intelligence
cd saas-revenue-intelligence
python -m venv venv
venv\Scripts\activate        # Windows
# source venv/bin/activate   # Mac/Linux
pip install -r requirements.txt
```

Copy `.env.example` to `.env` and fill in your keys:

```bash
cp .env.example .env
```

Train the model (required before first run):

```bash
python -m model.weights
```

Start the server:

```bash
python server.py
```

## Model

Ridge regression fit on log-transformed features: `log(headcount)`, `log(open_roles)`, `rank_score`, `review_momentum`, `acv_norm`, and a `headcount × rank` interaction term. RidgeCV selects regularisation strength automatically via leave-one-out cross-validation.

Training excludes companies with ARR > $5B (Salesforce, Workday etc.) since they sit far outside the typical $50M–$3B query range and would skew the fit.

Retrain after adding rows to `training_data.json`:

```bash
python -m model.weights
```

Backfill headcount for new training rows (uses Crustdata, costs credits):

```bash
python backfill_headcount.py
```

## Signal Fallback Chain

Each signal degrades gracefully when sources are unavailable:

**Headcount:** Crustdata → PDL → LinkdAPI → Proxycurl → SerpAPI → ScraperAPI → LinkedIn direct → `0`

**Hiring:** Ashby → Greenhouse → Lever → Workable → BambooHR → Rippling → `0`

**Reviews:** G2 → Trustpilot → `empty` (Crustdata G2 data used as fallback in `server.py`)

**Traffic:** Tranco list downloaded once, disk-cached for 3 days in `.tranco/`

**Pricing:** Scraped pricing page → Claude extraction fallback → `unknown`

If all signals fail the model returns a Low-confidence estimate using whatever partial data is available rather than erroring.

## Known Limitations

- **Small companies (<100 employees):** model tends to over-predict by 2–3x. The confidence interval widens automatically when headcount is low.
- **Platform-hosted domains** (e.g. `notion.so`): Crustdata sometimes returns hosted-page records instead of the actual company. Domain validation in `headcount.py` handles most cases.
- **JS-rendered pricing pages:** the scraper uses plain HTTP, so pages requiring JavaScript to render prices return `unknown`.
- **Static model:** does not self-improve from usage. Retrain manually as new verified ARR data becomes available by adding rows to `training_data.json` and running `python -m model.weights`.
